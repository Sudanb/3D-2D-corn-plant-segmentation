# Data Preparation — PLY to Projected Image Pipeline

## Overview

Training images are synthetically generated by projecting 3D terrestrial laser scan (TLS) point clouds of field-grown maize plants onto a virtual camera using perspective projection. Each plant is rendered from 75 viewpoints, producing RGB images, semantic projections, and integer instance masks used for instance segmentation training.

## Input Data

| Source | Description |
|--------|-------------|
| `FielGrwon_ZeaMays_RawPCD_100k/` | Raw point clouds — scanner reflectance encoded as RGB (contrast-stretched) |
| `FielGrwon_ZeaMays_SegmentedPCD_100k/` | Segmented point clouds — each instance (stem, leaf1–leafN) assigned a unique RGB color label |

Both clouds share the same 3D geometry. The segmented cloud provides ground-truth instance identity per point.

## Coordinate Frame Preprocessing

Raw PLY files are in TLS scanner space (Z-down, roots toward +Z). Before projection:

1. **Centering** — subtract the raw cloud centroid from both clouds so the plant base is at the origin
2. **Z-flip** — negate Z coordinates so +Z points upward (canonical upright plant orientation)

Result: both clouds are aligned, centred, Z-up, ready for camera orbit.

## Camera Model

### Intrinsics — Intel RealSense D457 (simulated)

| Parameter | Value |
|-----------|-------|
| Image size | 640 × 640 px |
| Horizontal FOV | 90° |
| Vertical FOV | 65° |
| f_x | ≈ 614.86 px |
| f_y | ≈ 593.34 px |
| c_x, c_y | 320, 320 px |

Intrinsic matrix:

```
K = [[fx,  0, cx],
     [ 0, fy, cy],
     [ 0,  0,  1]]
```

### Extrinsics — Virtual Camera Orbit

Cameras are placed on a hemisphere around the plant origin. Parameters:

| Parameter | Values |
|-----------|--------|
| Azimuths | 15°, 60°, 135°, 225°, 315° (5 positions) |
| Elevations | −30°, −15°, 0°, +15°, +30° (5 positions) |
| Distances | 1.0 m, 1.25 m, 1.5 m (3 distances) |
| **Total views per plant** | **75** |

For each viewpoint, the extrinsic matrix is constructed as:

1. Camera position computed from spherical coordinates (azimuth, elevation, distance)
2. Orthonormal camera basis {x, y, z} built so the camera looks at the origin
3. Rotation matrix R assembled from basis vectors; translation t = −R · cam_pos
4. 4×4 transform T = [R | t; 0 | 1]

## Projection

For each viewpoint, 3D world points P are projected to pixel coordinates (u, v):

```
P_cam = R · P_world + t       # world → camera frame
[u', v', w] = K · P_cam       # camera → image plane (homogeneous)
u = u' / w,  v = v' / w       # normalise to pixel coords
```

Points outside the image boundary or behind the camera (w ≤ 0) are discarded.

## Output Images (per plant × per view)

| File | Content |
|------|---------|
| `rgb_VVV.png` | 640×640 uint8 BGR — raw cloud colors, depth-sorted (painter's algorithm: far→near) |
| `seg_VVV.png` | 640×640 RGB — segmented cloud colors; each pixel's color = instance label |
| `mask_VVV.png` | 640×640 uint16 — integer instance IDs: 0=background, 1=stem, 2..N=leaf1..leafN (ordered bottom→top by mean Z) |

## Densification (Optional)

Projected point clouds produce sparse, noisy images at large distances. A morphological densification step fills gaps:

1. Morphological closing (3×3 kernel) on the filled pixel silhouette to identify interior gaps
2. Distance transform to find the nearest valid pixel for each gap
3. Colors and instance IDs propagated to gap pixels via nearest-neighbour lookup — leaf instance boundaries are preserved

## Instance Ordering

Leaf instances are numbered bottom-to-top by mean Z height in the segmented cloud after Z-flip. This gives consistent ordering across views:
- Instance 1 → stem
- Instance 2 → leaf1 (lowest, oldest leaf)
- Instance N+1 → leafN (highest, youngest leaf)

Up to 16 leaf instances are labelled (leaf1–leaf16).

## Dataset Split

Plants are split at the plant level (not image level) to prevent data leakage:

| Split | Plants | Images (×75 views) |
|-------|--------|-------------------|
| Train | 275 | 20,625 |
| Val | 59 | 4,425 |
| Test | 59 | 4,425 |
| **Total** | **393** | **29,475** |

Split uses random seed 3 for reproducibility.

## Format Conversion

`training_format.py` converts the per-plant image directories into training-ready formats:

### COCO JSON (`instances_{train|val|test}.json`)
- Polygon segmentation per instance (extracted from `mask_VVV.png` contours)
- Bounding boxes derived from polygons
- Category IDs: 1-indexed (1=stem, 2–17=leaf1–leaf16)

### YOLO Segmentation (`.txt` per image)
- Normalised polygon coordinates: `class_id x1 y1 x2 y2 ...` (values in [0, 1])
- One line per instance

### Directory Structure
```
training_data_fmt/
├── images/
│   ├── train/   plant_XXXX_rgb_VVV.png
│   ├── val/
│   └── test/
├── labels/
│   ├── train/   plant_XXXX_rgb_VVV.txt   (YOLO format)
│   ├── val/
│   └── test/
└── annotations/
    ├── instances_train.json   (COCO format)
    ├── instances_val.json
    └── instances_test.json
```

## Class Configurations

Two versions of the formatted dataset were prepared:

| Directory | Classes | nc |
|-----------|---------|----|
| `training_data_fmt/` | stem, leaf (all leaves merged) | 2 |
| `training_data_fmt_17/` | stem, leaf1–leaf16 (individual) | 17 |

The 2-class version collapses all leaf instances into a single *leaf* category at format conversion time; the underlying `mask_VVV.png` files are identical for both.
